Introduction to Definitions

Node Definition Basics

The attributes of a node definition (<nodedef>) and it's child parameters are shown in the UI interface. All attributes have lowercase string identifiers. This UI interface is the template used for documentation for each node definition.

Every node definition must have a unique name.

For the "core" libraries shipping with MaterialX a "ND_" prefix is used as a prefix with the the follow proposed syntax template to provide long term uniqueness:

ND_<node category>_<type>_<version>

While it is possible to change this name, this can cause issues with dependents that reference this name directly. This includes references for node instances which are stored in a document.

Each node definition must have a node attribute. This is the unique identifier to indicate the category of a node instance. One or more node definitions may use the same category, though each must have a unique "signature" (unique set of input, output, and token parameters).

An output type is also required. Different definitions for the same category can have the same or differing output types. If there is more than one output parameter on a definition then the type is multioutput. This this case the individual outputs need to be examined to determine their respective types. Two or more definitions with the same category may have a "multioutput" type.

It is worthwhile to have a version identifier for very definition to avoid complications trying to add this in later on. A recommended version number is either "1.0" , or a number matching the version of a given external specification. For example versions for "USD" definitions start with "2.x".

The isdefaultversion attribute should be provided at the same time as it determines the definition to instantiate when creating a nodes instance (when on explicit version set indicated). Versioning is also importer to allow for "upgrade" logic support.

It is possible to indicate that a definition inherits it's interface from another definition by specifying an inherit value. This value should be set to a definition "name". Inheritance allows for specialization as well as for handling version changes. For example the standard surface and USD texture definitions use inheritance.

Though not strictly required it is very useful to have some description of the definition. This is specified in the doc attribute of the definition wherein all text is assumed to be in UTF-8 format.

A node group nodegroup is optional but should be specified if it has a semantic which is used for evaluation including shader code generation. This includes:

  • "texture" Texture lookups test for this group identifier for example to perform convolutions
  • "adjustment"Used for heuristic to test for transparency
  • "conditional"Will cause branching in shader and re-generation is required for different conditions
  • "organization"Does not perform any computation
  • "translation"Is a shading model translator


Node Definition Implementations

A definition can be implemented either in code or using a functional node graph. This can be dependent on the output shading API "target".

The association between a node definition and the graph can either be specified via the nodedef attribute on a node graph or via an <implementation> association as of version 1.38.4.

The recommended functional graph name should start with "NG_" and be of the form:

NG_<node category>_<type>_<version>

The following diagram shows the two possible ways implementations can be associated with definitions. As of 1.38.4 the <implementation> approach is recommended as it the same way that code associations are specified.

Node Definition Parameters

The input and output parameters shown in the UI template include all the possible properties and value variants whether they are specified or not.

The required attributes are:

  • name The name must be unique between all parameters on the definition.
  • typeThe parameter type
Optional attributes
  • doc: Description of the parameter. It is recommended to add this info user understanding
  • uniform (input): If specified the input data is uniform (vs varying) Only inputs can have this specified.
  • enumvalues and enum: The set of possible values and their corresponding labels respectively.
  • value: The default value for inputs. Depending on the type, values may be qualified by additional attributes:
    • unit and unittype: For floats and vectors. Defines the unit and unit type for the input (e.g. "cm" and "distance"). Either one or the other may be specified
    • colorspace: For color3 and color4 and filename types. Defines the input color space. Note that a value of "none" means that this is a non-physical color and should not undergo any colorspace transforms
    • defaultgeomprop: String identifier of a default geometry stream to route as input
  • default: The default value of an output when a node instance is disabled (The disable attribute is set true)
  • defaultinput: The value on a definition's input to use by default value when a node instance is disabled.
Optional UI Attributes for Inputs
  • uimin, uisoftmin Hard and soft UI minimum value
  • uimax, uisoftmax Hard and soft UI maximum value
  • uistep UI step size value
  • uifolder Can be used to group parameters in the UI
  • uiadvanced Considered to be for "advanced" users


API methods

To retrieve information the API classes are as follows:

  • NodeDef: node definition
  • Implementation: nodedef definition implementation
  • NodeGraph: node graph implementation
  • Input: interface and node inputs for definitions and node graph implementation nodes
  • Output: outputs for definitions, node graphs and nodes

For brevity, only essential API calls shown in the UI template. Otherwise the attribute names or labels are used to denote where the information resides.

Input and output ports can be inherited from a parent definition. Only the non-inherited ports are shown per node definition. Note that in MaterialX terminology, getting "active" information means to get inherited information. Corresponding APIs having an "Active" string as part of the API name.

To get input and output attribute information the follow method is available: getAttribute(<attribute name>)

The general method to get value information (marked as 'value' in the template) is: getValue()

The API calls have the same name for all supported APIs (Python, Javascript, C++)
