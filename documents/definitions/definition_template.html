Node Definitions

Node Definition Basics

The attributes of a node definition (<nodedef>) and it's child parameters are shown in the UI interface. All attributes have lowercase string identifiers. This UI interface is the template used for documentation for each node definition.

Every node definition must have a unique name. For the "core" libraries shipping with MaterialX a "ND_" prefix is used as a prefix with the the follow proposed syntax template to provide long term uniqueness: 

    ND_<node category>_<type>_<version>
While it is possible to change this name, this can cause numerous issues with dependents that reference this name. Even within MaterialX usage a node can be instantiated using this name as the identifier of what to create. Files / assets saved using this name could be hard to maintain if the name is changed.

Each node definition must have a node category attribute. This is the unique identifier used to determine the category of node to created. 1 or more node definitions may use the same category, but each must have a unique interface defined by it's input, output, and token ports.

A type is also required. This is the output type for the node definition. Different definitions for the same category can have the same or differing output types. If there is more than one output in a definition then the type is multioutput, in which case the individual outputs need to be examined to determine their respective type.

It is worthwhile to give every definition a version identifier to avoid complications trying to add this in later on. A recommended version number is either "getValue()" , or a number matching the version of a given specification. For example versions for "USD" definitions start with "2.x". The isdefaultversion attribute should be provided at the same time as it determines the definition to instantiate when creating a nodes instance without specifying an explicit version. Versioning is also importer to allow for "upgrade" logic support. It is possible to indicate that a node definition inherits it's properties from another node definition by specifying an inherit attribute value. This can be used for specialization and also for versioning where only some specific propertie(s) have changed between versions. For example the "standard_surface" and "USD texture" definitions use inheritance.

Though not strictly required it is very useful to have some description of the definition. This is accessible in the doc attribute of the definition. It is assumed that all text is in UTF-8 format.

A node group nodegroup is optional and is used when performing code generation to determine how to handle a node. This is especially important to tag whether a definition is a texture lookup.

The API class for a definition is: NodeDef. Matching API calls to retrieve the information is written in Python format. In general the API calls for all supports APIs use the same name (Python, Javascript, C++)


Node Definition Implementations

A definition can be implemented either in code or using a functional node graph depending on the output shading API "target". The association between a node definition and the graph can either be specified via the nodedef attribute on a node graph or via an <implementation%gt; association as of version 1.38.4. The recommended functional graph name should be of the form: 

    NG_<node category>_<type>_<version>

For the node definition documentation, diagrams of this graphs are generated as Mermaid graphs


Node Definition Parameters

The input and output parameters show in the UI template show all the possible properties.

Note that parameters can be inherited from a parent definition. There is a different query to get all parameters instead of just those defined on the derived definition. This type of query is termed to be getting "active" information with the APIs having an "Active" string as part of the API name.

The required attributes are:

  • name The name must be unique between all parameters on the definition.
  • type
Optional attributes
  • doc Description of the parameter. It is recommended to add this info user understanding
  • uniform (input)Only input types can be specified to be a uniform or varying
  • enum and enumvalues The possible set of values enumerated and mapping to a value which matches to the type.
  • valueThe default value for inputs. Depending on the type values may be qualified by additional attributes:
    • unit and unittype For floats and vectors. Defines the unit and unit type for the input (e.g. "cm" and "distance"). Either one or the other may be specified
    • colorspace For color3 and color4 and filename types. Defines the input color space. Note that a value of "none" means that this is a non-physical color and should not undergo and colorspace transforms
    • defaultgeompropString identifier of a default geometry stream to route as input
  • default To default value of an output when a node instance is disabled ("disable" is set True)
  • defaultinput The value on a definition input to use as the defautl value when a node instance is disabled
Optional UI Attributes for Inputs
  • uimin, uisoftmin Hard and soft UI minimum
  • uimax, uisoftmax Hard and soft UI maximum
  • uistep UI step size
  • uifolder Can be used to group parameters in the UI
  • uiadvanced Considered to be for "advanced" users
