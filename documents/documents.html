Documents

This area is under construction
Properties

Documents are top level containers of all 'elements' supported by MaterialX. This can be roughly split up into two distinct categories:

  • Materials and Shader Graphs: This includes all nodes, definitions, implementations and shader translation
  • Geometric binding: This includes looks, material assignments, and application bindings
There is no organization or partition of elements though separation of content into the two categories is generally useful, especially as bindings overlap other systems such as USD.

Some important properties that are maintained at the document level include:

  1. Version: This is the MaterialX release version is independent definition versions
  2. Namespace : This can be used to partition content. Namespaces must be handled properly when dealing with multiple documents at runtime.
  3. Colorspace : This can be used to define the working color space, but as with namespaces care must be handled carefully when dealing with multiple document usage at runtime.
  4. Source URI : References to where content originated from can be specified here, and is useful as a partitioning mechanism.

Document Import

Single Document Model

Even though any number of documents may be kept in files for interopability, a single "working" document is generally required at runtime. All main interfaces work under the premise that:

  • content is cumulative (appended)
  • some validity checks are performed at read time
  • duplicate elements are disallowed and thus skipped. Incomplete reading can thus occur.
  • non-categorized nodes are invalid. i.e. There is now way to preserve nodes which are of "unknown" category, and again reading can be "lossy". What this practically means is that at a minimum a node definition is required to read into the document before-hand.
Interfaces
  • Reading interfaces: Though MaterialX has been built to allow any format to perform reading the core distribution comes with XML format support. Little validity checking is performed here through duplicate checking is enforced.
  • Copy interfaces: These interfaces may be used at the child level for various purposes. Via the document interface, this performs a replacement.
  • Import interfaces: This is currently specialized for reading in node definition libraries and include files, and performs per element copying into the target main document.

    For documents which have namespaces, these namespaces are essentially "flattened" so that individual elements are namespace qualifiers added to their identifiers. The semantic API naming to indicate that an API supports namespaces is "qualified". For example, 'getQualifiedName()' vs 'getName()'.'

Currently the only "inclusion" mechanism is via XML include directives.

File Referencing / File "Collections"

As there is no current concept of one document referencing another document at runtime, it can be hard to determine where content originates from once "merged" into the main document. One key way to keep track of where content comes from is the fact at import always "tags" imported elements with their original URI. (sourceURI). This can be useful for a various workflows including exclusion of these elements at write time. An example can be to exclude library definitions from node instances. At runtime, these tags can be used for instance to enforce additional user rules such as being non-editable, or non-deletable.

For node definitions, there is currently no formal construct for a "collection" (set) of documents to be used as a "library" / "module" / "package" etc and is thus up to the user to overlay semantics to partition content.
